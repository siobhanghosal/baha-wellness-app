# Stakeholder Feedback Preparation

## Recommended Session Framing

Present the BAHA prototype as:

- an architecture-backed demonstration
- a partial implementation of the intended product
- a tool for validating flow clarity, trust, and adoption concerns

Do not present it as:

- production-ready software
- clinically validated workflow software
- a finalized privacy or consent implementation

## Audience-Specific Framing

| Audience | Focus | Avoid overclaiming |
| --- | --- | --- |
| Clinicians | escalation, boundaries, consent, safe support framing | chatbot scope, clinical automation |
| Teachers | referral ease, classroom practicality, trend clarity | unrestricted access to student data |
| Parents | trust, summaries, consent, data rights | precision of analytics or persistence model |
| NGO partners | reach, adaptability, multilingual readiness | localization completeness today |
| Investors | system coherence, roadmap credibility, implementation path | current prototype fidelity |

## Pre-Session Checklist

1. Reset prototype state.
2. Use fictionalized persona data only.
3. Enter through the demo launcher, not deep links.
4. Hide reviewer-only controls unless actively needed.
5. Prepare a spoken disclaimer on prototype scope and privacy limitations.

## Suggested Feedback Questions

1. Which part of the flow felt most trustworthy?
2. Which step felt ambiguous or too generic?
3. Did role boundaries feel clear?
4. Did any screen imply capabilities you would expect BAHA to explain more carefully?
5. What evidence would you need before trusting this in practice?

## Feedback Export Guidance

- label every session with audience type and date
- remove identity-like details before sharing externally
- separate product feedback from clinical or privacy concerns
- track any issue that maps to PRR-001 through PRR-008 as a release blocker
